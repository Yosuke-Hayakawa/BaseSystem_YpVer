2.1.2 
Test Group 
Test Group はグループ分けすることが目的ですので、実装ロジックである開始処理と終了処理
（Preparation やCompletion）を実装することはできません。 
Test Group の中にもさらにTest Group を実装することが可能ですので、開始処理や終了処理が必要でな
い単なるグループ分けが目的の際には、Test Group を使用してテスト構成ごとに分類することが可能です。 
 
 
 
 
 
 
 
 
2.1.3 
Test Fixture 
Test Fixture は、Test Case だけでなく、Test Sequence も実装できます。 
Test Sequence とTest Case が含まれているTest Fixture において、共通の開始処理と終了処理
（Preparation とCompletion）を定義できます。 Test Fixture に含まれるTest Sequence とTest 
Case は、CANoe 上のテスト実行ツリーに個別に表示されます。 
また、Test Fixture の中にも複数Test Fixture を実装できますが、コマンドを実装することはできません。

2.1.4 
Test Sequence 
Test Sequence は、コマンドを含めることができ、Test Case と同様、Test Sequence は、開始処理と終了
処理（Preparation とCompletion）を含めることができます。 
 
一方で、このTest Sequence 内のTest Case はCANoe 上の実行ツリーには個別に表示されないので、
Test Case 単位での実行のアクティブ化、および非アクティブ化はできません。

Case ごとに実装できるので、状態遷移させての評価を行うことができます。

2.2 ファイルの再利用 
2.2.1 
ファイルリンク 
Project View 内で、ドラッグ＆ドロップ操作によりTest Unit のテストファイルを他のTest Unit にリンクさせ
ることができます。 
たとえば、Test Case List を複数のTest Unit で使用したい場合、Test Case を管理するのは1 箇所で良
く、使用する場合は、パラメーター値を変えるだけで簡単に再利用することができます。 
（Test Case List に関しては、2.8 節「Test Case List 機能」をご参照ください。） 
 
 
 
 
リンクされたファイルは、アイコンが変化します。 
 
 
リンクを追加すると、アイコンが通常のTest Table ではなく、リンク専用のアイコンになります。これは、対象
のTest Unit 直下にファイルが作成されているのではなく、ファイルを参照しているということです。つまり、こ
のテストファイルを編集するということは参照元のファイルを編集することになります。

2.2.2 
Home Directory とLibraries 
現在作成中のプロジェクトファイルが存在するディレクトリをHome Directory と呼び、通常Project View
の下に配置されています。そして、Libraries はTest Project とは切り離されたファイルを参照、定義するこ
とが出来ます。これは、別のプロジェクトで作成していたテストファイルを別のプロジェクトで再利用する場合
に使用できます。 
※ファイルを参照する場合に上書きされたくない場合には、2.2.4 項「読み取り専用として使用」をご参照くだ
さい。 
 
① vTESTstudio でLibrary の設定を行ないます。リボンタブ[Layout] – [Views]グループ内の
「Libraries」を選択し、Libraries View を表示させてください。 
 
 
 
② Libraries View 内で右クリック[New Library]を選択します。すると、Libraries のファイルを保存する
先を選択します。新規に作成する場合は、Test Project と同じフォルダで問題ありません。別のプロ
ジェクトのファイルを参照したい場合は、そのファイルのパスを指定してください。 
 
 
 
③ 登録されたフォルダを右クリック[Create] - [Test Table]を選択します。Test Table 名は任意の名前
を付けてください

2.2.3 
Shareable Folders 
テストファイルのリンクの他に、複数のテストファイルをまとめたフォルダをリンクして使用することが可能です。
このフォルダをShareable Folders と呼びます。これは、フォルダごとリンクすることができるので、共有可能
なファイルとフォルダが視覚的に明確になり、リンクした場合でも分かりやすくなります。 
また、Shareable Folders は、ファイルへのリンクを含むことができるのでライブラリーよりもテストユニットに
近くなります。 
 
 
ユースケースとしては、複数のテストユニットで共通のテスト（たとえば、初期設定や起動処理など）がある場
合に、Shareable Folders を作成し、その中にテストファイルとパラメーターを格納して、フォルダごとリンクさ
せることでリンクさせたファイルなどが視覚的に明確になります。 
 
① ここでは、事前に作成してあった初期化テストと起動テストの2 つをそれぞれ項目ごとに分けて
Shareable Folder を作成し、リンクさせる方法を記述します。まず、プロジェクトにShareable Folder
を作成します。対象のプロジェクトの上で右クリックし、[New Shareable Folder]を選択します。 
 
 
 
② 今回は2 つのShareable Folder を使用するので、先ほど追加した[Shareable Folders]内にもう1 つ
Shareable Folder を作成します。作成後、識別できるように名前を付けておきます。

2.2.4 
読み取り専用として使用 
別のプロジェクトで作成したファイルをLibraries から再利用する場合、リンク先のファイルを編集した場合、
リンク元のファイルまで書き換えられてしまいます。複数プロジェクトで使用するようなファイルなどは、ファイ
ル自体を[読み取り専用]とすることで対応できます。 
 
① たとえば、以下のような状況において、Shareable Folder にLibraries から[Parm_Init]がリンクされ
ています。このままの状態で、Shareable Folder 側のパラメーターファイルの数値を変更した場合、リ
ンク元のパラメーターファイルの値まで変更されてしまいます。 
 
 
 
 
② 変更したくないファイルなどは、ファイルを読み取り専用とすることで編集不可の設定にすることができま
す。今回の場合ですと、Libraries の参照元フォルダを開き、対象ファイルを選択し、右クリックし[プロパ
ティ（R）]を選択します。

データ駆動とキーワード駆動によるテストアプローチには、以下のメリットがあります。 
 
テストスクリプトを再利用できるため、作成済みのテストスクリプトと類似のテストを作成する際に効率的 
 
プログラミング言語に慣れていないテスト担当者でも容易にテストケースの作成が可能で、人的リソース

上記を実現する前提として、スキルを持った方が事前にテストスクリプトを作り込み、ライブラリー化しておく必

照ください。 
 
まず、データ駆動はJSTQB シラバス1では以下のように記載されています。 
 
「テスト入力（テストデータ）をスプレッドシートに入れるなどして分類する。そして、入力データを読んだり、異

する。スクリプト言語に慣れていないテスト担当者でも、キーワードを使ってテストを定義できる。キーワード駆
動によって、テストをテスト対象のアプリケーション用に調整することができる。」 
 
このキーワードとは、たとえば「電源ON」のようなテスト動作を示します。キーワード駆動ではこのような動作

ば、テストケースが増えても対応し易くなります。 
 
このキーワードに相当するスクリプトを作成する人は、テスト対象システムについての知識と、テストシステム
に関する知識の両方が求められます。（例：ECU の機能や動作に関する知識と、CANoe やVTsystem な
どテストシステムを要求通りに動作させるための知識） 
 
一方、テストケースを作成する人に必要な知識はテスト対象システムについての知識だけとなり、人的リソー
スを有効活用することができます。

3.5.5 
ログファイル設定 
ログ取得をテストトリガーとした場合、フィールドコードを設定しなければログファイルが上書きされてしまい、
期待のログファイルを取得することができません。 
 
ログファイルの設定は、[測定設定]内のログファイル設定で行います。 
 
 
 
ログファイル設定を開くと、ログファイルの出力先とファイル名を設定します。出力先と出力するファイル名は
任意で決定してください。 
 
 
 
ここで、テストトリガーを使用する場合は[フィールドコード]から[各トリガーでインクリメント]か[トリガー名にイ
ンクリメント]を選択してください。 
[トリガー名にインクリメント]の方が、Test Case 等の名前がログファイル名になりますので判別が容易です。
