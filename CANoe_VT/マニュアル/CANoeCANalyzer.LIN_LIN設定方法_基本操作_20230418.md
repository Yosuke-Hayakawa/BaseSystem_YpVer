LDF の準備 
CAN 通信のデータベースはCANdb ファイル(*.dbc)ですが、LIN 通信のデータベースはLDF(*.ldf)となり
ます。LDF には、マスターノード、スレーブノードの設定やスケジュール設定も含まれているため、LIN 通信の
解析およびシミュレーションを行ううえで非常に重要です。 
CANoe/CANalyzer.LIN を使用したLIN 通信シミュレーション（ヘッダー／レスポンスの送信）の設定は、
"LDF を割り付けのみ"です。（CAPL プログラミングは不要） 
CANoe/CANalyzer.LIN では、専用のエディターである"LDF Explorer" を用いてLDF を簡単に作成できま
す。使用方法についてはアドバンストマニュアル "CANoe/CANalyzer LDF Explorer" を参照してください。

CANoe/CANalyzer.LIN によるLDF の設定方法 
LDF をCANoe/CANalyzer.LIN に割り付けることにより、解析およびシミュレーション環境を簡単に構築するこ
とができます。LDF で定義されたすべてのマスター／スレーブノードをシミュレーションできるCANoe.LIN に対
し、CANalyzer.LIN は任意の1 ノードのみシミュレーションできます（※各チャンネル１ノードではなく、全チャン
ネル合わせて１ノードのみ）。

CANoe.LIN で新規コンフィギュレーションを作成し、LDF を設定します。 
 
<操作手順>

① "使用可能なテンプレート" から [LIN]を選択 
② [コンフィギュレーションを生成]を押下 
 
(2) [シミュレーション]タブ – [シミュレーション設定]を選択 
 
① [データベース]を右クリックし、[追加]を選択 
② 使用するLDF を選択

CANoe/CANalyzer.LIN がLIN マスターを模擬する場合、「マスターモード」と「マスターレジスタ」が有効に
なっている必要があります。これらが有効になっていない場合、CANoe/CANalyzer.LIN からLIN ヘッダーが
送信されません。一方、CANoe/CANalyzer.LIN がLIN スレーブを模擬する場合は逆にこれらの設定を無効
とする必要があります。 
なお、CANoe の場合は複数ノードを模擬できるため、模擬するノードにマスターノードが含まれている場合は
「マスターモード」と「マスターレジスタ」を有効にしてください。CANoe がLIN スレーブのみを模擬する場合に
は、「マスターモード」と「マスターレジスタ」を無効にしてください。 
 
＜LIN マスターを模擬する設定＞ 
■データベース設定を使う設定 
 
 
上記の「データベース設定を使う」が有効になっている場合、他の設定（LIN ISC やLIN マスターノードのアク
ティブ／非アクティブなど）から推測して自動的にマスターモードの切り替えが行われます。そのため、多くの場
合この「データベース設定を使う」が有効の状態で意図どおりの動作となります。一方、以下に挙げる例の場合
のように意図と異なって動作してしまうこともあります。意図どおりの動作とならない場合には、「データベース設
定を使う」を無効にし、「マスターモード」と「マスターレジスタ」を明示的に設定してください。 
 
例：CANoe 上のLIN ネットワークに配置されたLIN ISC およびLIN マスターノードが共に非アクティブとした
うえで、CAPL 等でLIN ヘッダーを送信する場合。CANoe は内部的にLIN マスターを模擬するノードがいな
いとみなしてしまい、「データベース設定を使う」を有効にしているとLIN ヘッダーが送信されなくなります。 
 
■データベース設定を使わず、「マスターモード」と「マスターレジスタ」が明示的に有効になっている設定 
⇒CANoe/CANalyzer.LIN がLIN マスターを模擬する場合の設定

の分流を防ぐことができます。

バスでの測定を実行できます。

6.5 CAPL プログラミング （CANoe.LIN のみ ※CANalyzer.LIN は機能制限あり） 
CAPL（Communication Access Programming Language）は、システム開発やソフトウェアモジュー

7.2 リプレイブロックを用いたログファイルの再生 
リプレイブロックを用いてログファイルの内容を再生することができます。これにより、ログファイルの詳細な解析や、
実車で測定したデータなどを評価対象ECU に刺激として与えることができます。 
 
なお、CANoe/CANalyzer.LIN にてリプレイブロックを用いたログファイルの再生を行う場合、CANoe または
CANalyzer はマスターを模擬している必要があります。つまり、リプレイブロックではマスターを含むノードのレスポ
ンスを再生することはできますが、マスターを含まない、スレーブノードのレスポンスだけを再生することはできません。 
LIN はプロトコルの特性上、LIN マスターがヘッダーを送信しないとデータを送信することができません。一方、リプレ
イブロックはログファイルのタイミングでフレームを送信する機能であるため、CANoe/CANalyzer.LIN がマスター
を模擬し、ヘッダーを送信できる状態でないと利用できません。そのため、LIN スレーブのスレーブレスポンスだけを
リプレイブロックで再生することができません。 
LIN スレーブのスレーブレスポンスだけを再生したい場合にはCANoe.LIN をお使いいただいたうえで、本書7.3 節
のシグナルジェネレーターを用いた方法を利用してください。 
 
リプレイブロックを用いていて、CANoe/CANalyzer.LIN からヘッダーが送信されない場合は、本書4.3 節に記載
のネットワークハードウェアの設定にて「マスターモード」と「マスターレジスタ」を明示的に有効にする設定を行ってく

TIPS ～リプレイ対象のフレームの任意設定～ 
「マスターのレスポンスのみを再生」はLDF の定義を基に再生されます。LDF 上でマスターから送信され
るフレームすべてをリプレイ対象とします。これを利用してLDF を編集することで、リプレイ対象のフレーム
を任意の範囲に設定することができます。 
たとえばスレーブECU の実機に対し、実車で測定したデータのうち該当するECU 以外のレスポンスを刺
激として与えたい場合を考えます。LDF 上、マスター、スレーブ1、スレーブ2 が存在していたとし、対象の
ECU をスレーブ1 とします。このとき、リプレイ用のLDF を作成し、スレーブ2 が送信するレスポンスをす
べてマスターから送信するよう定義を変更します。これにより、リプレイブロックからマスター、スレーブ2 の
レスポンスが送信されるようになります。なお、LDF が修正されたため、解析、シミュレーションなどでは元
のLDF の場合と異なる動作となります。ご注意ください。

■スタートビットの妨害について 
LIN 妨害ブロックや、同様の妨害機能を持つCAPL 関数（linInvertHeaderBit、linInvertRespBit など）を
用いてヘッダーやレスポンスのスタートビットを妨害することはできません。 
これはLIN 仕様上、インターバイトスペース等の時間の不確定性があるため、スタートビットの開始タイミング
が事前には分からないためです。スタートビットがデータの開始を示すため、この後ろのビット（データビットや
ストップビット）については狙って妨害をかけることが可能ですが、前述のとおりスタートビットについては狙って
妨害することが難しいため、CANoe.LIN では対応していません。

10.5 LIN Stress Feature Set（CAPL 関数） 
LIN Stress Feature Set はLINstress 機能のCAPL 関数です。LIN ストレスIG / LIN 妨害ブロックで妨害可能
なLINstress 機能はLIN Stress Feature Set（CAPL 関数）で対応できます。 
また、LIN Stress Feature Set のみ対応しているLINstress 機能もあります。 
 
LIN Stress Feature Set（CAPL 関数）は、以下のケースで有効です。 
• 
手動操作では対応できない精密なタイミングで妨害印加が必要な場合 
  例: シグナル値の変更時などの特定イベント発生時に、一定時間妨害を印加 
• 
CANoe テスト機能を用いたLIN バス妨害テストを実装／構築する場合 
  例: 実機が送信するすべてのレスポンスフレームに自動で妨害印加し、結果をレポート出力 
•

12.1.1 CANalzyer.LIN で複数LIN チャンネルにフレーム送信できますか？ 
残念ながらできません。CANalyzer はコンフィギュレーション上に複数のLIN チャンネルを持っている場合でも、全
体として1 つのノードのみの模擬が可能です。したがいまして、1 チャンネルにのみフレーム送信が可能です。複数
LIN チャンネルに対してフレーム送信が必要な場合には、CANoe.LIN をご利用ください。 
 
12.1.2 CANoe/CANalyzer.LIN からLIN ヘッダーが送信されません 
本書４章の設定が正しく行われているかご確認ください。特に見落としやすい点として、4.3 節のネットワークハード
ウェア設定の「マスターモード」と「マスターレジスタ」の有効化の設定と、CANalyzer の場合は模擬するノードがマス
ターノードになっているか、という点が挙げられます。詳細は4 章を参照してください。 
 
12.1.3 CANoe/CANalyzer.LIN からスレーブレスポンスが送信されません 
本書４章の設定が正しく行われているかご確認ください。CANoe.LIN でスレーブノードのみを模擬をする場合や
CANalyzer.LIN でスレーブ模擬をする場合、4.3 節のネットワークハードウェア設定の「マスターモード」と「マスター
レジスタ」が無効となっている必要があります。また、CANalyzer.LIN の場合は模擬するノードが正しく設定されてい
るかについてもご確認ください。LIN ISC を利用してスレーブレスポンスを送信する場合も、これらの設定が必要です。
これらの設定方法の詳細は4 章を参照してください。 
 
12.1.4 リプレイブロックでログを再生すると少し時間がずれます 
リプレイブロックのログ再生ではトレースWindow のTime 列に記録された時間をもとにマスターからヘッダーが送
信されます。このTime 列の時間とは、正常フレームの場合はチェックサムフィールドのストップビット（フレームの最
終ビット）となり、エラーのある場合はCANoe がエラーと判定した時間が打刻されます。そのため、ログ取得時の最
終ビットがリプレイ時のSOF（フレームの開始タイミング）になり、そのTime 差は約1 フレーム時間（+処理時間）と
なります。 
ログファイルに正常フレームだけが含まれる場合、基本的には1 フレーム時間分程度のずれがすべてのフレームに
発生するため、ログ再生時のフレーム同士の時間のずれはあまり大きくありません。一方、例えばレスポンスが送信
されないTransmError などのエラーが発生していると、CANoe がエラーと判定するまでに待機時間が必要となる
ため、ログに打刻される時間が正常フレームよりも遅くなり、ログ再生時にエラーのあるフレームと前後の正常フレー
ムとの間で多少のずれが発生します。

CANoe/CANalyzer はLIN 仕様に準拠し、LIN フレーム内の未使用ビットはリセッシブに設定して送信されます。
これをドミナントに変更する設定等は無いため、回避方法として、LDF 上で該当の未使用ビット領域にダミーのシグ
ナルを定義し、すべてドミナントとなるよう初期値を設定することで、実際上の未使用ビットについてドミナントで送信さ
せることができます。 
 
12.1.13 LIN フレームの未使用ビットがドミナントになっていることを確認したい 
目視で確認するのであれば、トレースWindow にてECU から送信されたLIN フレームの未使用領域のデータを
確認する方法があります。 
CANoe をご利用でテスト機能にて未使用ビットの確認をする場合には、前項にて記述した回避方法を利用すること
ができます。つまり、LDF 上で該当の未使用ビット領域にダミーのシグナルを定義し、このシグナルの値を自動テスト
などで確認する、という方法となります。 
 
 
12.2 CANoe.LIN のLINstress 機能に関するお問い合わせ 
12.2.1 バス上を流れるデータに対してビット反転等の妨害をかけたい 
CANoe が受信する（＝バス上に送信される）信号を妨害する場合には、LIN 妨害機能（LIN 妨害ブロック、または
CAPL の妨害機能セット）をご利用ください。このLIN 妨害機能の詳細は、本書の10.4 節、10.5 節を参照してくださ
い。 
 
12.2.2 CANoe が送信するフレームにビット反転を加えると、後続のデータが送信されません 
CANoe は送信時にエラーを検出した際、次のバイト境界で後続バイトの送信を停止します。これはLIN 仕様に準拠
した動作となります。CANoe からECU に対して、異常なデータを伴いつつ後続のデータも送信したい場合には、
linSendVariableBitStream 関数などのビットストリーム出力を行う関数をご利用ください。なお、ビットストリーム出
力を行う関数は上記以外にもいくつかありますので、ご所望の動作に相応しい関数をCANoe ヘルプよりお選びくだ
さい。 
 
12.2.3 LIN 妨害ブロックでスタートビットを妨害できますか？ 
残念ながらできません。スタートビットはLIN 仕様上、開始タイミングが事前にわからないため、狙って妨害すること
ができません。詳細は本書10.4.2 項の最下段を参照してください。 
 
12.2.4 1/16 ビット時間単位でビット反転すると、波形がなまります 
1/16 ビット時間単位でのビット反転など、急峻なエッジの立ち上がり／立ち下がりが必要な場合は、フラッシュモード
を有効にしてから行ってください。フラッシュモードのON/OFF はlinActivateFlashMode 関数にて制御できます。
詳細は本書10.5 節に記載していますので、参照してください。
