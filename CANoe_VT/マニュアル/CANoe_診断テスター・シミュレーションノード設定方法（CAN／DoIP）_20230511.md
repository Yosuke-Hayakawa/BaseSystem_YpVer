1.4 本書で紹介するCANoe 診断テスター・シミュレーションノードの機能 
診断コンソール、フォールトメモリ、セッション制御などのCANoe 診断機能セット（下記図の Diagnostic 
Windows）は、常に内蔵診断チャネル（下記図の CANoe/CANalyzer "Built-in" Diagnostic Channel）を
使用します。 
 
テストモジュール、テストユニットは、内蔵診断チャネルだけでなく、CAPL コールバックインターフェイス（CCI）と、
対応する転送プロトコル（TP）DLL を用いた機能も使用できます。 
CCI を使用することで、診断レイヤーとTP レイヤーを相互に接続するような機能で、TP レベルのフォールトイン
ジェクションを含むテストや複雑なシミュレーションを実現できます。 
また、シミュレーションノードは、CANoe Ver9.0 以前はCCI が必須ですが、CANoe Ver10.0 以降では、CCI
だけでなく、内蔵診断チャネルも使用できます。 
 
本書では、"内蔵診断チャネルを用いた診断シミュレーション環境" の構築方法を記載します。一方、CCI につ
いても本書７章で紹介していますので、CCI をご利用の方は7 章を参照してください。

1.4.2 
診断シミュレーションノード 
CANoe Ver10.0 では、簡単にCAN/LIN/FlexRay/K-Line/Ethernet 通信を用いたECU の診断シミュレー
ションが可能な機能（シミュレーションノード用の内蔵診断チャネル）が追加されました。 
本機能を使用することで、CANoe を用いた "診断テスターの動作確認" が簡単に実施できます。 
 
本書では、本機能（内蔵診断チャネル）を使用した診断シミュレーションノードの作成方法を記載します。 
 
＜シミュレーションノード用の内蔵診断チャネルの特長＞ 
 
必要な設定は、診断データベースファイル（CDD, PDX, MDX ファイル）と対象ノードの選択のみ 
 
デフォルト設定でECU シミュレーションが可能 （CAPL プログラムは不要） 
・ 
診断データベースに定義された診断サービスを受信した場合、ポジティブレスポンスを自動送信 
・ 
診断データベースに未定義の診断サービスを受信した場合、ネガティブレスポンスを自動送信 
 
デフォルト設定とは異なるシミュレーションを実施したい場合のみ、CAPL プログラムで実装 
・ 
パラメーター値を設定したポジティブレスポンスの送信、ネガティブレスポンスの送信、など 
 
（旧バージョンでは必要な）コールバック関数（CAPL コールバックインターフェイス（CCI））が不要 
 
（旧バージョンと比較し、）CAPL プログラムのコード数が削減 
 
 
 
旧バージョンのCANoe を動作させることが必要なコンフィギュレーション、あるいはTP レベルのフォールトイ
ンジェクションを含むテストを実行することが必要な場合、あるいはさまざまなネットワーク間の診断ゲートウェ
イのような "より複雑なECU シミュレーション" に対処することが必要な場合は、CCI を使用する必要があり
ます。詳細は、アプリケーションノート "診断ツールとしてのCANoe およびCANalyzer"（1.2節） を参照して
ください。

void MainTest () 
{ 
  // テストケースTest1()呼び出し 
  Test1(); 
} 
 
TestCase Test1() 
{ 
  diagRequest Door.EcuIdentification_Read diagReq; 
 
  // リクエスト送信 
  diagSendRequest( diagReq); 
  if (TestWaitForDiagRequestSent(diagReq, 2000)== 1) // リクエスト送信完了まで待機 
  { 
    TestStepPass("Request was sent successfully!"); 
  } 
  else 
  { 
    TestStepFail("Request could not be sent!"); 
    return; 
  } 
 
  // レスポンス受信チェック 
  switch( TestWaitForDiagResponse( diagReq, 200)) // レスポンス受信まで待機 
  { 
  case 0: // タイムアウト: 200ms以内にレスポンス受信無し 
    write("No answer from ECU!"); 
    TestStepFail("Read ID", "No answer from ECU!"); 
    break; 
  case 1: // レスポンス受信 
    TestReportWriteDiagResponse(diagReq); // テストレポートにレスポンス情報を出力 
    if( diagGetLastResponseCode(diagReq) == -1) 
    { 
      // ポジティブレスポンス受信 
      write("RecordDataIdentifier: %d", (long)diagGetRespParameter(diagReq," RecordDataIdentifier")); 
      TestStepPass("Read ID", "Positiv response received!"); 
    } 
    else 
    { 
      // ネガティブレスポンス受信 
      write( " EcuIdentification_Read failed: 0x%x", diagGetLastResponseCode( diagReq)); 
      TestStepFail("Read ID", "Negative response received"); 
    } 
    break; 
  default: // 内部/設定エラー 
    TestStepFail("Read ID", "Error in TestCase! Verdict unreliable."); 
  } 
}

5.6 受信した診断リクエストを解析して送信レスポンスを切り替える場合 
"on diagRequest *" イベントは、"on diagRequest" イベントに未定義の診断リクエストを受信した際の処
理を実装できます。これにより、CDD ファイルに未定義の診断オブジェクト受信時の動作を定義できます。 
下記例では、受信した診断リクエストから、サービスID を取得し、それぞれ異なるレスポンスを送信します。 
 
サービスID:0x01 
 レスポンスサイズを3 バイトに変更後、2 バイト目に0xFF を格納してポジティブレスポンスを送信 
 
サービスID:0x02 
 レスポンスサイズを5 バイトに変更後、4 バイト目に0xFF を格納してポジティブレスポンスを送信 
 
その他のサービスID 
 NRC に0x12 を格納してネガティブレスポンスを送信 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
※"on diagRequest *" イベントを定義した場合、CAPL で定義していないすべての診断リクエスト受信時
に本イベントが呼び出されます。そのため、デフォルト設定で実行される"ポジティブレスポンスの自動送信
"は行われませんので注意してください。下記例の様に、各診断オブジェクトに対し、ポジティブレスポンス
の送信処理を定義／実装する必要があります。（DefaultSession_Start, TesterPresent_Send） 
 
 
 
 
 
 
 
 
 
 
// 上記以外のリクエスト受信時の対応 
on diagRequest * 
{ 
  diagResponse * undefResp; 
  byte ServiceId; 
 
  // ダイアグオブジェクト初期化 
  DiagInitialize(undefResp); 
 
  ServiceId = DiagGetPrimitiveByte(this,0); 
  DiagSetPrimitiveByte(undefResp, 0, ServiceId + 0x40); 
 
  switch(ServiceId){ 
    case 0x01: 
      // ポジティブレスポンスサイズ変更（レスポンスサイズを3バイトに変更） 
      diagResize(undefResp, 3); 
      // ポジティブレスポンスデータ値設定（2バイト目に0xFF設定） 
      DiagSetPrimitiveByte(undefResp, 2, 0xFF); 
      // ポジティブレスポンス送信 
      diagSendResponse(undefResp); 
      break; 
    case 0x02: 
      // ポジティブレスポンスサイズ変更（レスポンスサイズを5バイトに変更） 
      diagResize(undefResp, 5); 
      // ポジティブレスポンスデータ値設定（4バイト目に0xFF設定） 
      DiagSetPrimitiveByte(undefResp, 4, 0xFF); 
      // ポジティブレスポンス送信 
      diagSendResponse(undefResp); 
      break; 
    default: 
      // ネガティブレスポンス送信 
      diagSendNegativeResponse(undefResp, 0x12); 
      break; 
  } 
} 
// Door.DefaultSession_Start(10 01) 
// ポジティブレスポンス送信 
on diagRequest Door.DefaultSession_Start 
{ 
  diagResponse this resp; 
   
  // ポジティブレスポンス送信 
  diagSendResponse(resp); 
} 
// Door.TesterPresent_Send(3E 00) 
// ポジティブレスポンス送信 
on diagRequest Door.TesterPresent_Send 
{ 
  diagResponse this resp; 
   
  // ポジティブレスポンス送信 
  diagSendResponse(resp);

8.9 CANoe が自動送信するCF を遅延・欠落させたい 
CF の最小送信間隔はSTmin という、診断仕様で定められたTP 層のタイミングパラメーターにより決定されま
す。CANoe の「診断ISO/TP 設定」の「トランスポートレイヤー」画面では、STmin を設定可能ですが、この設
定値はCANoe がFC フレームを送信するときFC フレームに記載されるSTmin の値として利用されるもので、
CANoe がCF を送信する際にはECU 側から送信されたFC フレーム記載のSTmin の値が利用されます。 
CAN TP 用CAPL 関数"CanTpSetSTmin"では動的にSTmin を設定できますが、この関数も上記のGUI
での設定と同様CANoe が送信するFC フレームに記載される値を設定するだけで、CANoe が送信するCF
の遅延には利用できません。（なお、STmin は仕様上127ms が最大となります。そのためこれらの設定や関数
でも設定できる最大値は127ms となります。STmin の詳細は診断仕様書を参照してください。） 
 
CANoe が自動送信するCF に対して任意の遅延時間を設定する場合には、いくつか方法があります。以下に
それぞれの利用するCAPL 関数と概要を記載します。この中で最も柔軟に設定可能なのは最後の方法となり、
この方法の場合はCF の欠落も可能です。この方法に関してはサンプルコンフィギュレーションも準備している
ため、最後に紹介する方法が推奨となります。なお、下記のCAPL 関数の利用にはCAN TP 用CAPL 関数
（osek_tp.dll）、CAPL コールバックインターフェイス（CCI）を使用する必要があります。CCI の詳細は、7 章を
参照してください。 
 
 
CanTpSetMinimumSTmin 関数を利用する 
CanTpSetMinimumSTmin関数はその名称通り、最小のSTminを設定する関数です。この関数単体で
CANoe が送信するすべてのCF の最小のSTmin を設定できます。なお、この関数で設定した値よりも大
きな値がECU からのFC フレームにて指定された場合は、FC フレーム指定の値が利用されます。 
例1: CanTpSetMinimumSTmin 関数で30ms を指定し、ECU からのFC フレームでは20ms が指定 
⇒CANoe が送信するCF はすべて30ms の送信間隔で送信 
例2: CanTpSetMinimumSTmin 関数で10ms を指定し、ECU からのFC フレームでは20ms が指定 
⇒CANoe が送信するCF はすべて20ms の送信間隔で送信 
 
ただし、127ms より大きな値は設定できず、仮に大きな値を設定しても自動的に127ms に設定されます。

variables 
{

{

8.18 診断通信のログを保存したり再生したりできますか？ 
可能です。診断通信のログは.asc や.blf などのフレーム形式で取得してください。また、これらのログファイルを

8.19 診断パラメーターをシグナル形式のログファイルに保存できますか？ 
残念ながら診断パラメーターはシグナル形式のログファイルに対応していないため、.asc や.blf などのフレーム
形式のログファイルに診断通信のフレームを保存していただく必要があります。なお、これらのフレーム形式の
ログファイルを、診断データベースの割り当てられたCANoe にて再生すると、診断データベースに基づく解釈
をログデータに付与しながら解析できますので、シグナル形式で保存しなくとも、ログデータ内の診断パラメー
ターの解析が可能です。

CANoe がCF を送信する場合、FC フレームに記載のSTmin を最小の送信間隔として送信を試みます。これ

8.37 フォールトメモリーWindow 画面にエラーが表示され、DTC を取得できない 
以下のようなエラーは、受信したレスポンスのフォーマット等に異常があり、DTC 情報を正しく読み取ることがで
きない場合に表示されます。 
 
 
この種のエラーの場合、診断データベースの定義と実際のECU からのレスポンスで差異が発生している可能
性があります。CANoe は診断データベースファイルの定義に従って情報を解釈しようと試みますので、受信し
たECU からのレスポンスがこの定義から外れたフォーマットだったり、あり得ない値だったりする場合、この種
のエラーが発生します。 
実際に受信したレスポンスはトレースWindow にて確認することができますので、以下のようにトレース
Window 上の受信データと診断データベースファイルの定義に差異がないか確認してください。 
 
 
診断通信ではレスポンスの長さを可変長に定めることも可能ですが、サービスによってどのようなフォーマットと
なるかは変わってきます。特にDTC 情報を読み出す場合、DTC の個数によってレスポンスの長さは可変とな
る一方、可変とはいってもどのような長さでも許されるわけではなく、DTC 情報は1 つにつき4byte ずつ記載す
ることがUDS の仕様で定められています。利用している診断データベースの定義次第ではありますが、多くの
場合UDS 仕様を踏襲していると考えられるため、ECU からのレスポンスの長さがこの仕様に合致しているか
確認してください。 
DTC の読み出しの場合、一般的には、最初にSID などの3byte の後に、DTC 個数×4byte の長さとなるた
め、レスポンスの長さは7byte,11byte,15byte,19byte などが取り得る値となります。ここから外れた値の場合
はフォーマット違反となり、上記のエラーの原因となっている可能性があります。上記のトレースWindow の表
示では、緑色で表示されたポジティブレスポンスの「DLC」カラムの値が14 となり、許された値から外れているこ
とが分かります。 
そのため、このようなエラーが出た場合には、診断データベースの定義と、ECU の実装を確認してください。

9.6 CANoe でのDoIP 通信（シミュレーション／解析機能）の制限

// on key [1] : Alive Check Request送信 
on key '1' 
{ 
  byte payload[1]={0x10}; // Dummy Byte 
  DoIP_TCPSend(0x0007, payload, 0); // Payload Type 7, zero bytes payload (Alive Check Request) 
} 
on ethernetPacket * 
{ 
  word payloadType; 
  byte data; 
   
  if(this.tcp.IsAvailable() == 1 && this.doip.IsAvailable() == 1) 
  { 
    // TCP & DoIPフレーム受信 
    write("swapWord( this.doip.word(2)=[0x%X]",swapWord( this.doip.word(2))); 
    payloadType = swapWord( this.doip.word(2)); // Payload Typeを取得 
     
    if(payloadType == 0x0006) // Routing Activation Response 
    { 
      write("Routing Activation response received"); 
      //Access the first byte of the DoIP frame 
      data = this.doip.byte(0); 
    } 
  } 
}

9.16 DoIP の動作について、標準と異なる通信を実行したい 
DoIP でも本書７章に記載のCCI を利用できますので、多くの場合CCI の利用により標準と異なる通信が実装

CCI を利用しない他の方法として、TCP/IP CAPL 関数を利用してDoIP 通信を実装することもできます。この
方法の場合はCCI よりも更に柔軟に実装できる一方、実装工数がかかります。この方法にて実装したサンプル
コンフィギュレーション「診断テスト_TcpIpApi を利用したDoIP 通信テスト」が弊社Web サイトの「製品機能活
用マニュアル」に掲載されていますので、TCP/IP CAPL 関数でDoIP 通信を実装したい場合には参照してくだ
さい。「製品機能活用マニュアル」のアクセス方法は、1.1 節を参照してください。 
 
9.17 DoIP.ini ファイルまたはCAPL で送信ポートを指定しているが、意図通りに動作しない 
DoIP やDoSoAd 通信にて、テスター側の送信ポートを、本書9.4 節に記載のDoIP.ini ファイルまたはCAPL
