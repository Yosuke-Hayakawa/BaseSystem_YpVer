FDX の概要 
リアルタイムデータ 
交換 
CANoe FDX (Fast Data eXchange) は、CANoe と他のシステム間でEthernet 接続を介
したシンプルで高速のリアルタイムデータ交換のためのプロトコルです。このプロトコルによ
り、他のシステムは、CANoe システム変数、環境変数およびバスメッセージの内容へアク
セスを行うことができます。 
 
他のシステムとは、たとえば、テストベンチのHIL システムやCANoe からのデータを表示
するために使用するコンピューターなどです。以下の章では、これらのことをHIL システムと
呼びます。

2.1 
プロトコルの概要 
2.1.1 UDP またはTCP をベースとした送受信 
UDP プロトコル 
FDX プロトコルは、UDP プロトコル (IPv4 またはIPv6) がベースになっています。UDP は、
広く普及している標準プロトコルで、HIL システムでも広く使用されています。  
CANoe とHIL システム間のデータ交換は、UDP データグラムの送受信によって実現され
ます。原則的には、HIL システムが最初にCANoe へデータグラムを送信するため、
CANoe がFDX プロトコルの通信で使用するIP アドレスとポート番号を把握している必要
があります。CANoe は、送信側のIP アドレスとポート番号を確認し、データを要求した送信
元に要求されたデータを返信します。 
TCP プロトコル 
リアルタイム性よりもデータ送受信の信頼性が必要となるケースでは、トランスポート層として
UPD の代わりにTCP を使用することもできます。この場合、CANoe はTCP ポートを開き、
HIL システムからの受信接続をリッスンします。接続を確立した後のデータ送受信はUDP の
場合と同じように行われ、FDX データグラムがTCP コネクション上で転送されます。 
トランスポート層と 
ポート番号 
デフォルトでは、CANoe は、FDX プロトコル用にポート2809 でUDP/IPv4 を使用する設定
になっています。トランスポート層とポート番号は、CANoe の [オプション] ダイアログ ([ファ
イル] - [オプション] - [拡張設定] - [XIL API & FDX プロトコル]) で変更できます。 
UDP およびTCP のデータグラムの詳しい構造は、セクション「2.2 プロトコルの説明」を参
照してください。データグラムの構造はシンプルなので、HIL システムでIP スタックが利用
可能で、デベロッパーに経験があれば、HIL システムにFDX プロトコルを実装するために
掛かる期間としては1～2 日程度が想定されます。 
  
2.1.2 データグループとデータ型 
データグループ 
メッセージの内容と変数は、データ交換のためグループとしてあらかじめ定義されており、
CANoe とHIL システムの間では、常にグループ単位で交換されます。グループにすること
により、シグナル項目ごとに識別子を使用する必要がないため、データをより効率よく格納
することができます。グループは、個別のファイルに定義します (セクション「2.3 FDX 記述
ファイル」を参照)。また、実際のデータ交換は、DataExchange コマンドによって実行され
ます (セクション「2.2.6 DataExchange コマンド」を参照)。 
データ型 
ネットワーク経由でのデータ転送で必要となる帯域を抑えるため、FDX プロトコルでは、
様々なデータ型を利用したデータ交換が可能です。HIL 側でのデータアクセスを容易にする
ため、データ型は、一般的なC/C++コンパイラおよび他のプログラミング言語で直接使用で
きるものに限られています。

バイトオーダー 
プロトコルバージョン1.2 まで、データ型はx86 アーキテクチャでエンコードされます (整数
値はリトルエンディアンをバイトシーケンスに使用、符号付き整数型は2 の補数を使用、浮
動小数点値はIEEE 形式を使用)。プロトコルバージョン 2.0 ではエンコーディングの設定を
変更することができます。データグラムヘッダー (セクション2.2.1 を参照) の1 ビットでバイ
トオーダーを指定し、リトルエンディアンまたはビッグエンディアンを指定できます。 
 
文字列は、1 バイトのASCII 文字で構成され、必ずNull バイトで終了します。これは、C プ
ログラミング言語の文字列形式に対応します。 
配列 
配列データ型のbytearray、floatarray、doublearray、int32array は、最初の4 バイトに配
列が使用するデータバイト数を示す値 (Uint32) を含み、その後に実際のデータバイトが続
きます。 
サイズ 
String、bytearray、floatarray、doublearray、int32array のデータ型のデータグループを定
義する場合、FDX 記述ファイルに文字列または配列の最大長を指定する必要があります。
文字列の長さには末尾のNull バイトが含まれ、配列 (bytearray、floatarray、doublearray
およびint32array) の場合は、データバイト数の4 バイトが含まれます。 
オフセット 
データグループを定義する場合、各シグナル/変数に固定のオフセット (バイト単位の値) が
割り当てられます。シグナル/変数がメモリレイアウト上で重複しないようにしてください。 
 
データグループ内でシグナル/変数に固定オフセットを使用できるようにするには、文字列お
よび配列が常に、データ送信時に (データグループの定義に基づく) 最大バイト数を取る必
要があります。未使用のバイトは、0 値で初期化されます。 
  
2.1.3 FreeRunning モード 
データグループの 
周期的送信 
CANoe では、明示的な要求に対する応答として (セクション「2.2.5 DataRequest コマンド」
を参照)、もしくはFreeRunning モードが設定されていることにより、HIL システムにデータ
グループを送信します。FreeRunning モードでは、データグループを定期的に送信する
か、またはCAPL 関数FDXTriggerDataGroup の呼び出しがトリガーになります。また、
データ送信は、測定開始直前および測定終了時にも実行できます。 
 
FreeRunning モードは、FreeRunningRequest コマンドを使用して設定します (セクショ
ン「2.2.8 FreeRunningRequest コマンド)」を参照)。CANoe では常に、この要求コマンドを
発行したUDP またはTCP のアドレスにデータを送信します。

シグネチャー 
シグネチャーは、データグラムが実際にCANoe を対象としているかどうか (あるいは、プロ
グラムがデータグラムを誤ってCANoe に送信したかどうか) をチェックするマジッククッキー

バージョン番号 
2 桁のバージョン番号は、互換性をチェックするために使用されます。同じメジャーバージョ
ン番号を持つ新しいプロトコルバージョンには、追加の情報が含まれることがありますが、
同じメジャーバージョン番号を持つ旧バージョンのプログラムでは、新しい情報を除いた
データグラムを処理できます。プロトコルは、メジャーバージョン番号が変更されるまで、互
換性を維持します。

データグラムをFDX クライアントに送信する場合、このクライアントから最後に受信した
データグラムがプロトコルメジャーバージョン1 を使用している場合は、CANoe はプロトコル
バージョン1.2 を使用します。このクライアントから最後に受信したデータグラムがプロトコ
ルメジャーバージョン2 を使用している場合、CANoe はプロトコルバージョン2.1 を使用し
ます。

fdxProtocolFlags フィールドの最下位ビットで、このダイアログで使用するバイトオーダー
を指定します。ビッグエンディアンを使用する場合はビットを1 に設定し、リトルエンディアン
を使用する場合はビットを0 に設定します。ビッグエンディアンに対応するにはプロトコル
バージョン2.0 以降が必要です。プロトコルバージョン1.2 以前を使用する場合、このビット
は0 にします。CANoe でデータグラムをFDX クライアントに送信する場合は、必ずこのク
ライアントから最後に受信したデータグラムで使用されているものと同じバイトオーダーを使
用します。FDX クライアントに2 つ以上のバイトオーダーを実装する必要はありません。

2.2.8 FreeRunningRequest コマンド 
FreeRunning モード
の切替 
FreeRunningRequest コマンドは、データグループのFreeRunning モードをオンにします。
このコマンドは、測定の開始前に、HIL システムからCANoe へ送信されることがあります。 
 
FreeRunning モードでは、CANoe からHIL システムへ自律的にデータを送信します。
データ交換は、開始準備段階 (PreStart)、測定終了時、または測定実行中に周期的に行っ
たり、CAPL 関数を呼び出してトリガーしたりすることができます。flags フィールドを使用す
ると、1 つ以上のオプションを選択できます。周期的送信の場合、データを最初に送信する
時期は、firstDuration フィールドによって決まります。FreeRunning モードが測定開始前
にすでに設定されている場合、firstDuration は、測定開始からデータグループの最初の
送信までの期間となります。測定の実行中にFreeRunningRequest コマンドを受信した
場合、firstDuration で指定された時間後、送信を開始します。 
 
CANoe での測定終了時に、すべてのFreeRunningRequests は削除されます。また、
FreeRunningCancel コマンドを使用すると、測定実行中にデータグループの
FreeRunning モードを明示的に終了できます。 
  
 
FreeRunning モードがすでにデータグループで起動しており、データグループに追加の
FreeRunningRequest コマンドをCANoe から受信した場合、この要求は、すでにアクティ
ブになっている要求と合わせて処理されます。

データグループ 
データグループは、フレーム、シグナル、PDU、システム変数、環境変数を記述する項目の
リストから成ります。groupID 属性は、データグループの一意のインターフェイス識別子とな

データバイト数を指定します (セクション「2.2.6 DataExchange コマンド」を参照)。

</datagroup>

列またはバイト配列の最大長となります (セクション「2.1.2 データグループとデータ型」を参

シグナル 
シグナルは、以下のように記述されます。 
  
 
 
<signal name="..." msg="..." database="..." node="..." 
        bus="..." channel="..." 
        direction="auto/txrq" value="phys/raw" /> 
 
 
 
シグナルは、名前 (name 属性)、送信されるメッセージ名 (msg 属性)、およびデータベース
名 (database 属性) で指定されます。 
 送信ノード 
送信ノード (node 属性) は、複数ノードからの同一メッセージ送信がプロトコル (J1939, 
NMEA 2000, GM LAN) で許可されている場合にのみ指定する必要があります。 
 バス 
バス (bus 属性) またはチャンネル (channel 属性) は、CANoe でデータベースが複数の
バスに割り当てられている場合にのみ指定する必要があります。この場合、CANoe シミュ
レーション設定でバス名を指定することも、バスに割り当てるチャンネルを指定することもで
きます (CAN1、CAN2、LIN1 など)。 
 生値/物理値 
value 属性は、データフィールドにシグナルの生値 (value=“raw”) または物理値 
(value=“phys”) のどちらを書き込むかを指定します。 
 方向 
CANoe では、すべてのシグナルに2 つのタイムスタンプ値が格納されています。1 つは、
バス上で正しく転送された時点の値 (Rx 値とも呼ばれます) で、もう1 つは、バス上で
CANoe から送信された時点の値 (TxRq 値) です。Rx 値は、ここで説明したインターフェイ
スを介して読込みのみ可能ですが、TxRq 値は、読み込んだり、設定することができます。
ただし、通常は、外部HIL システムでのみ設定します。direction 属性は、読み込む値を識
別するために使用します。direction=“auto”に設定すると、Rx 値をCANoe → HIL でデー
タ交換を行い、TxRq 値をHIL → CANoe データ交換で行うように設定されます。
direction=“txrq”に設定すると、TxRq はどちらの場合でもアクセスできます。 
 
システム変数 
sysvar は、システム変数またはその構造体メンバーあるいは配列メンバーを記述するとき
に使用します。名前 (name 属性) 以外に、ネームスペースを指定するnamespace 属性も
含み、物理/生メンバーアクセスのvalue 属性 (シグナルエレメントの属性に相当) をサポー
トします。システム変数構造体を使用する場合は、項目のデータ型は “bytearray” にする
必要があります。構造体メンバーアクセスを指定する構文は、C プログラミング言語と類似
しています: “variable.member”。配列エレメントにアクセスする場合は、C と同等の角括
弧表記法を使用します: “variable[index]”。メンバーおよび配列エレメントの指定は、混在
させたり、連結させたりできます (多次元配列や構造体の配列の場合など)。 
value 属性は任意で、“raw” がデフォルト値になります。 
  
 
 
<sysvar name="..." namespace="..." value="phys/raw"/> 
 
 
 
環境変数 
環境変数は、envvar エレメントを使用して指定します。name 属性は、環境変数の名前と
なります。 
  
 
 
<envvar name="..."/> 
 
 
 
PDU 
PDU (AUTOSAR PDU) はFlexRay バスシステムのプロトコルデータユニットを記述するた
めに使用されます。PDU は名前 (name 属性) で識別されます。名前が一意でない場合、
追加の属性 (database、bus、channel、node) を使用して、PDU を一意に識別できます。 
 
PDU に使用できるのは、バイト配列データ型のみです。バイト配列の長さはPDU の長さに
等しくなければなりません。バイト数を含む4 バイトは計算に含まれません。

CAN メッセージはframe エレメントで記述できます。メッセージは名前 (name 属性) で識別

は独立して行われます。最後に受信したメッセージの内容が送信されます。メッセージが受
信されていない場合、長さ0 のバイト配列が送信されます。

関数呼び出し

の指定と同様に、ポートのpath 属性を指定する必要があります。この場合、コンシュー
マー側のファンクションポートまたはサービスメソッドポートを指定する必要があります。さら
に、グループ識別子に似た数値識別子を属性callID に指定します。関数呼び出しおよびそ
の応答用としてデータグラム内に予約されるバイト数はそれぞれ、属性callSize と
returnSize に指定します。オプションのrepresentation 属性を使用すると、パラメーター
データをRaw, Impl 値とPhys 値のいずれで転送するかを指定できます。既知の
identifier サブエレメント以外に、function エレメントの子はありません。 
  
 
 
<function callID="…" callSize="…" returnSize="…" path="…"/>

2.4 
シリアル化 
シリアル化 
Data Exchange コマンドおよびFunction Call コマンドのペイロードは、固定バイナリーレイ
アウトを持たない複合データタイプの値 (オプションのメンバーを持つ構造体、可変長の配
列など) を含むことができます。この場合、値をコマンドのバイト配列のメンバーごと、または
エレメントごとにシリアル化して、変数の情報を保持する必要があります。FDX 記述が特定
の表示を要求した場合、まず値がその表示に変換されてからシリアル化されます。選択した
表示がPhys である場合、すべての複合データ型を再度メンバーごとにシリアル化する必
要があります。これは、物理値の表示に、データグラムにコピー可能なネイティブのバイナ
リー表示がないためです。 
Impl またはRaw 表示の複合データタイプ値はすべて、変換されることなくメモリーから単
純にデータグラムにコピーされます。つまり、特にバイト順は適用されません。CANoe は、
そのような値をリトルエンディアンのバイト順で受信し、また値をそのバイト順で送信すること
を要求します。 
  
2.4.1 プリミティブのタイプ 
プリミティブのタイプ 
プリミティブのデータタイプはすべて、定義されたメモリーレイアウトを使用してシリアル化さ
れます (セクション「2.1.2 データグループとデータ型」を参照)。値がそれ自体でデータグ
ループ項目の場合、その生値、impl 値、または物理値が、その項目に指定されたデータタ
イプに変換されてからデータグラムに出力されます。プリミティブの値が単体の関数呼び出
しパラメーター、またはシリアル化される複合データタイプのメンバーである場合、親の複合
データタイプまたは関数プロトタイプで定義されたデータタイプを使用して、その生値、impl
値、または物理値がデータグラムに直接出力されます。 
  
2.4.2 文字列 
文字列 
値エンティティまたは関数呼び出しパラメーターの文字列値は、他の文字列値とは異なり、
UTF-8 でエンコードされます。そのような文字列はすべて、そのUTF-8 バイト配列をデータ
グラムにコピーすることによりシリアル化されます。これは、0x00 から0x7F までの範囲内
の文字のみを使用している限り、ASCII 文字列と互換性があります。 
  
2.4.3 結合 
結合 
固定のバイナリーレイアウトを持たない結合は、まずディスクリミネーターの値を出力するこ
とによりシリアル化されます。ディスクリミネーターとは、現在有効な結合メンバーのインデッ
クス (宣言順) を指定する16 ビット符号付き整数です。現在有効な結合メンバーがない場
合、値-1 が出力されます。それ以外の場合、ディスクリミネーターの値の後に、これらの規
則に従ってシリアル化された、現在有効な結合メンバーが続きます。

2.4.4 構造体 
構造体 
固定のバイナリーレイアウトを持たない構造体は、宣言順でメンバーごとにシリアル化され
ます。各メンバーは、これらの規則に従ってシリアル化されます。メンバーがオプションであ
る場合、まずオプションのメンバーの値が存在するかどうか (存在する場合は1、存在しな
い場合は0) を示す1 バイトフラグがデータグラムに出力されます。値が存在する場合はフ
ラグの直後に現れます。  
  
2.4.5 ジェネリック配列 
ジェネリック配列 
固定バイナリーレイアウトを持たない配列は、まず現在の配列長を符号なし32 ビット整数
値としてデータグラムに出力し、次に規則に従って照準にエレメントされた配列エレメントが
続きます。

3.1 
概要 
帯域 
セクション「3.1.2 ネットワーク負荷」に、大部分のアプリケーションは100MBit/s のEthernet
通信で十分であることが説明されています。近年の一般的なコンピューターおよびベクター
のVN8911 デバイスには1GBit/s Ethernet インターフェイスが装備されています。このた
め、十分なEthernet 通信の帯域が確保されています。 
 
CANoe のFDX インターフェイスのパフォーマンスは、主として使用しているCPU の処理
能力によって決まります。データはEthernet インターフェイスを介して転送されるだけでは
なく、CANoe で処理される必要があります。同時に、CANoe は接続されているバスシステ
ム (CAN、LIN、FlexRay など) 上のメッセージを処理し、残りのバスシミュレーションも実行
する必要があります。そのため、FDX を介して交換できるデータの量は、残りのバスシミュ
レーションの程度やCPU の処理能力に依ります。

関数呼び出し 
TesterAPI 
この例では、関数呼び出しコマンドの使用方法を示します。 
CANoe FDX プロトコルの関数呼び出しコマンドは、CANoe 内部の分散オブジェクト (DO) 
または通信オブジェクト (CO) のメソッドに対応したものとなっています。この例では、Vector 
Communication Description Language (vCDL) を使用して宣言され、CAPL コードとして
実装された分散オブジェクトを使用します。 
Add とSendRequest という2 つのメソッドを持つ、Foo という名前の分散オブジェクトを宣
言するvCDL ファイル: 
  
 
 
version 2.0; 
 
namespace TesterAPI 
{ 
  object Foo 
  { 
    internal uint32 Add(uint32 x, uint32 y); 
 
    [OneWay=true] 
    internal void SendRequest(uint32 id, uint32 flags, bytes 
payload);     
  } 
}  
 
 
  
Foo.SendRequest 
分散オブジェクトのメソッドを実装するCAPL コード: 
  
 
 
on fct_called TesterAPI::Foo.Add  
{      
  this.Result = this.x + this.y;  
}  
 
on fct_called TesterAPI::Foo.SendRequest 
{ 
  dword id; 
  dword flags; 
  byte  payload_data[64]; 
  dword payload_length; 
   
  long i; 
   
  flags = this.flags; 
  id = this.id; 
  payload_length = elcount(this.payload); 
  memcpy(payload_data, this.payload); 
   
  Write("TesterAPI::Foo.SendRequest: id=0x%X flags=0x%X, 
payload-length=%d", id, flags, payload_length);

FDX データベースでは、XML エレメントfunction の属性path が分散オブジェクトのメソッ

用されます。

callSize がFDX データベース内で意味を持たないため、この例では単純にゼロに設定さ
れます。
