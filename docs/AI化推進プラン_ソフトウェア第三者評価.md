# ソフトウェア第三者評価 AI化推進プラン

**作成日:** 2026年3月17日  
**対象部署:** ソフトウェア第三者評価チーム（11名）  
**目的:** GitHub Copilot マルチエージェントを活用し、評価業務の効率化・品質向上を図る

---

## 1. エグゼクティブサマリー

設計部署が「boss→elite→mob」のマルチエージェント体制を構築しているのと同様に、第三者評価チームも同構造のAI化を推進する。

**目指す姿:** 作業者は **bossにやりたいことを伝えるだけ** で、各業務ステップのドラフト成果物をAIが生成し、人間がレビュー・承認するフローに移行する。

## 3. マルチエージェント構成

```
ユーザ（作業者）
　│ やりたいことを自然言語で指示
　▼
boss   ← 作業者が唯一やり取りする相手
　│ 仕様確定・判断・タスク分解を指揮
　▼
elite
　│ タスク分解・担当割当・進捗管理
　│ 作業量に応じて必要な数のmobを起動
　├──────────┬────────────┬────────────┤
　▼          ▼            ▼            ▼
mob          mob          mob          mob ...
（仕様解析）  （VT環境）   （テスト仕様書） （仕様解析）
                                       ↑ 同じテンプレートで
                                         並列分担可能
```

mob は汎用ワーカー。固定の役割を持たず、elite がタスクごとに必要な数だけ起動し、タスクテンプレート（ドメイン知識・出力フォーマット）を渡して指示する。
作業量が多いステップでは、同じテンプレートで複数 mob を並列起動できる（例：仕様書10冊を3 mob で分担）。

## 4. タスクテンプレートと各エージェントの役割詳細

### 4.1 boss— 全体指揮

- **役割:** 作業者から指示を受け、業務全体を仕切る
- **主な仕事:**
  - 作業者の意図を確認し、仕様・AC（完了条件）を確定する
  - eliteにタスク分解を依頼する
  - 重要判断事項を作業者に「お伺い」形式で提示する
  - 最終成果物のレビューを実施する
- **入力:** 作業者のチャット指示、仕様書ファイル（PDF/Word/Excel）
- **出力:** タスクリスト、進捗ダッシュボード更新、最終レポート

### 4.2 elite— コーディネーター

- **役割:** bossの指示を受け、mobへのタスク割当と調整を行う
- **主な仕事:**
  - 業務ステップをサブタスクに分解する
  - 作業量に応じて必要な数の mob を起動し、適切なタスクテンプレートを渡す
  - 各mobの成果物を統合・取りまとめる
  - 進捗を `status/dashboard.md` に反映する
- **入力:** bossからの指示、仕様書
- **出力:** サブタスク一覧、統合ドラフト文書

### 4.3 タスクテンプレート：仕様解析

- **役割:** 仕様書群を解析し、テストに必要な情報を抽出・整理する
- **主な仕事:**
  - 客先仕様書・制御仕様書・通信仕様書等のポイントを抽出
  - 通信ビットアサインの一覧化
  - 機能安全要求（ASIL等）の抽出・整理
  - 試験対象機能のリストアップ
- **入力:** 各種仕様書（PDF/Word/Excel）
- **出力:** `output/spec_summary.md`（仕様サマリー）、 `output/signal_list.md`（信号一覧）

### 4.4 タスクテンプレート：VT環境

- **役割:** CANoe/VTシステム環境構築のドラフトを生成する
- **主な仕事:**
  - 通信仕様書・ビットアサインからDBCファイルの信号定義ドラフト生成
  - テスト手順からCAPLスクリプトのスケルトン（骨格）コード生成
  - PANEL設計のガイドライン提示
  - IO設定ドキュメントの作成支援
- **入力:** 通信仕様書、ビットアサイン表
- **出力:** `output/dbc_draft.md`（DBC定義ドラフト）、 `output/capl_skeleton.can`（CAPLスケルトン）

### 4.5 タスクテンプレート：テスト仕様書

- **役割:** 仕様解析結果をもとに、テスト設計仕様書のドラフトを作成する
- **主な仕事:**
  - テスト設計仕様書テンプレート（.xlsm）に則したドラフト内容の生成
  - 非機能試験（ノイズ印加、電源電圧変動）の試験観点列挙
  - ユースケース試験のシナリオ提案
  - テスト観点マトリクスの作成
- **入力:** `output/spec_summary.md`、仕様書サマリー、テンプレート構造情報
- **出力:** `output/test_spec_draft.md`（テスト仕様書ドラフト、Markdownで確認用）

### 4.6 タスクテンプレート：テストケース

- **役割:** テスト仕様書をもとに、テストケース一覧を生成する
- **主な仕事:**
  - テスト仕様書からvTESTstudio用テストケース一覧を生成
  - 入力値・期待値・試験手順の整理
  - 境界値・同値分割等、テスト設計技法の適用提案
  - テストステップのドラフト記述
- **入力:** `output/test_spec_draft.md`
- **出力:** `output/testcase_list.md`（テストケース一覧）

### 4.7 タスクテンプレート：結果解析

- **役割:** テスト実行後のレポートやログを解析し、NG内容を整理する
- **主な仕事:**
  - CANoeレポートの解析と判定結果のまとめ
  - NGケースの根本原因候補の列挙
  - ログデータからの異常値・パターン検出支援
  - NG内容の再現手順の整理
- **入力:** CANoeテストレポート（HTML/XML）、ログファイル（BLF/ASC）
- **出力:** `output/ng_analysis.md`（NG解析レポート）

### 4.8 タスクテンプレート：報告書

- **役割:** 懸念点確認シートと試験報告書のドラフトを生成する
- **主な仕事:**
  - NG解析結果から懸念点確認シートの記入内容を生成
  - 試験結果サマリーの作成（試験項目数、合否件数等）
  - 試験報告書（本文・サマリー）のドラフト生成
  - 設計部署への伝達事項の整理
- **入力:** `output/ng_analysis.md`、`output/testcase_list.md`
- **出力:** `output/concern_sheet_draft.md`（懸念点確認シートドラフト）、 `output/test_report_draft.md`（試験報告書ドラフト）

### Phase 1：パイロット導入（2〜3ヶ月）

- **対象業務:** ステップ1（仕様理解）とステップ4（テスト仕様書作成）に限定してパイロット実施
- **対象メンバー:** 先行チーム2〜3名
- **進め方:**
  1. 既存案件の仕様書をAIに読み込ませ、仕様サマリーを生成させる
  2. 生成されたサマリーをベテランが検証し、誤りや漏れを記録する
  3. テスト仕様書ドラフトを生成し、既存の完成品と比較評価する
  4. フィードバックをプロンプト・instructions に反映する
- **成功指標:** 仕様サマリー生成時間が従来比50%以上削減

### Phase 2：エージェント拡張（3〜5ヶ月）

- **対象業務:** ステップ5（テストケース作成）、ステップ7（結果解析）を追加
- **対象メンバー:** 全員へ展開開始
- **ポイント:**
  - CAPLスケルトン生成はエンジニアがレビューしてからvTESTstudioに取り込む
  - CANoeレポートのフォーマットをエージェントに学習させる
- **成功指標:** テストケース作成工数が従来比30%以上削減

### Phase 3：フル運用（5ヶ月以降）

- **対象業務:** ステップ3（VT環境）、ステップ8（懸念点シート）、ステップ9（報告書）を追加
- **対象メンバー:** 全員フル活用
- **目標:**
  - 作業者はbossへの指示とレビュー・承認のみを担当
  - 反復作業の80%以上をAIが初稿生成

## 7. 作業者（ユーザ）の操作フロー

Phase 3 完成後、作業者は以下の3ステップだけで業務を進められる。

```
Step 1: Copilot Chat で boss を選択し、依頼を投げる
         例: 「「○○製品のテスト仕様書を作りたい。仕様書を添付します。」

Step 2: bossが「お伺い」してきたら選択肢を選ぶ
         例: 「Bで / 推奨案で / Aで、ただし〇〇は△△に変えて」

Step 3: status/dashboard.md を見て成果物を承認する
         ↓
     output/ フォルダの各ドラフトをレビューして確認
```

### 業務別の指示テンプレート例

#### テスト仕様書作成を依頼する場合

```
以下の製品のテスト仕様書（ドラフト）を作成してほしい。

■ 概要
- 製品: 〈製品名・型番〉
- 対象試験: 非機能試験（ノイズ印加、電源電圧変動）、ユースケース試験
- 参照仕様書: 〈添付ファイル名を列挙〉

■ 制約
- テスト設計仕様書テンプレート（docs/テスト設計仕様書_テンプレート_20260206.xlsm）の構成に沿うこと
- 第三者評価の視点（ノイズ・電圧変動等の非機能試験重視）で作成すること
- 成果物は output/ 配下に Markdown 形式で出力すること

■ 期待する成果物
- output/spec_summary.md（仕様サマリー）
- output/test_spec_draft.md（テスト仕様書ドラフト）

■ 進め方
1. まず仕様のポイントと試験観点を整理して見せてください
2. 重要判断は私に確認してください
3. タスク分解→並列作成→レビュー→統合の流れで進めてください
```

#### テストケース作成を依頼する場合

```
output/test_spec_draft.md のテスト仕様書をもとに、テストケース一覧を作成してほしい。

■ 概要
- テスト設計技法: 境界値分析、同値分割、ユースケース記述を適用すること
- 入力値・期待値・試験手順を明記すること

■ 制約
- vTESTstudio での実装を想定した記述にすること
- 成果物は output/testcase_list.md に出力すること

■ 進め方
1. まずテスト観点の分類方針を見せてください
2. 重要判断は私に確認してください
3. タスク分解→並列作成→レビュー→統合の流れで進めてください
```

## 8. リスクと対策

| リスク | 影響度 | 対策 |
|---|---|---|
| **AI生成物の品質不足** | 高 | 必ず人間がレビュー・承認プロセスを経る。初期は2名ペアでレビュー |
| **仕様書の機密情報漏洩** | 高 | GitHub Enterprise / 組織内Copilotを使用。個人アカウントではなく組織アカウントで管理 |
| **ツール習熟の遅れ** | 中 | Phase 0 の教育を十分に実施。社内勉強会を月1回開催 |
| **CANoe/VT固有知識の不足** | 中 | KnowledgeBaseにCANoe/CAPL固有の知識をinstructionsとして蓄積 |
| **AI幻覚（ハルシネーション）** | 中 | 出力には必ず根拠となる仕様書の参照箇所を記載させる |
| **メンバーの抵抗感** | 低〜中 | 「AIは補助。判断はエンジニアが行う」という役割分担を明確にする |
